# Архитектура Historical Bloodlines

## 1. Тип приложения

Historical Bloodlines — пакетный преобразователь документов. Он не является
CRUD-системой и не хранит изменяемое состояние между запусками.

```text
входной Excel
    → нормализация данных
    → разрешение ссылок
    → domain-граф
    → проверка
    → геометрическая раскладка
    → печатный документ
```

По устройству pipeline ближе к компилятору или генератору отчётов, чем к
веб-приложению. Excel выполняет роль постоянного источника и пользовательского
редактора, поэтому промежуточная база данных не требуется.

## 2. Направление зависимостей

```text
CLI
 ↓
Application services
 ↓
Domain

Infrastructure → DTO / Domain
```

`BuildGenealogyUseCase` оркестрирует конкретный сценарий: чтение книги, parsing,
assembly, validation и rendering. Он не содержит алгоритмов разрешения имён или
построения линий.

## 3. DTO

### RawGenealogyRowDTO

Почти дословно представляет строку Excel. Поля с суффиксом `_raw` ещё не
интерпретированы. Например, `children_raw` остаётся строкой, а не коллекцией
людей.

### GenealogySheetDTO

Содержит:

- `name` — техническое имя вкладки;
- `display_title` — текст из служебной строки `Название`;
- `rows` — кортеж сырых строк.

Техническое имя используется для идентичности источника. Отображаемое название
можно исправлять без разрушения ссылок.

### PersonReferenceDTO

Описывает текстовую ссылку на потенциального человека:

- нормализованное имя;
- порядковый номер брака или ребёнка;
- исходное значение;
- уточнение в скобках.

На этом этапе ссылка ещё не является `Person`: parser не знает, к какому из
одноимённых людей она относится.

### ParsedGenealogyRowDTO

Нормализованное представление строки. Титулы становятся tuple, годы —
`ReignPeriod`, а дети и супруги — `PersonReferenceDTO`.

## 4. Domain-модель

### SourcePersonKey

```python
SourcePersonKey(sheet_name="Комнины", row_number=12)
```

Это исходная идентичность человека. Имя человека не используется как ключ,
поскольку в одной родословной могут встречаться одноимённые люди.

UUID вычисляется через UUIDv5 из `SourcePersonKey`, поэтому одинаковая книга
получает одинаковые идентификаторы при повторных запусках.

### Person

`Person` является entity: его отличает идентичность, а не только набор полей.
Он содержит собственные атрибуты, но не содержит вложенных списков `children`
или `spouses`.

### Relation objects

`MarriageRelation` хранит пару супругов.

`ParentChildRelation` хранит направленный индивидуальный факт
`родитель → ребёнок`.

`FamilyChildRelation` хранит визуально-семейный факт: от какого набора родителей
должна выходить линия к ребёнку.

Разделение необходимо, потому что ребёнок должен визуально выходить из
конкретного союза, а не автоматически из каждого супруга своего родителя.

### Genealogy

`Genealogy` — агрегат и in-memory source of truth:

```text
persons: dict[UUID, Person]
marriages: set[MarriageRelation]
parent_child_relations: set[ParentChildRelation]
family_child_relations: set[FamilyChildRelation]
```

Словарь даёт быстрый доступ по UUID. Множества дедуплицируют одинаковые связи.
Отдельный индекс нормализованных имён ускоряет поиск кандидатов resolver.

## 5. Parser

`GenealogyRowParser` выполняет только синтаксическую нормализацию:

- очищает пробелы;
- разделяет значения по `;` и переносам строк;
- извлекает `1)`, `2.` и другие порядковые маркеры;
- отделяет уточнение в скобках;
- строит согласованные периоды правления.

Parser не ищет людей и не создаёт relation objects.

## 6. Assembler и resolver

`GenealogyAssembler` использует двухпроходный алгоритм.

Первый проход создаёт всех `Person`. Благодаря этому ссылка из первой строки
может указывать на человека, который находится в конце листа.

Второй этап сначала разрешает браки, затем детей. Такой порядок позволяет не
принять одноимённого супруга за ребёнка.

`PersonReferenceResolver` фильтрует кандидатов по династии и нормализованному
имени, затем оценивает:

1. совпадение уточнения;
2. точное совпадение имени;
3. слова из исходной ссылки и титулов;
4. направление строки для ребёнка;
5. расстояние до исходной строки.

Если подходящей строки нет, assembler создаёт placeholder-person. Это позволяет
выводить терминальные подписи вроде «Графы Вермандуа».

## 7. Валидация NetworkX

`NetworkXGenealogyValidator` создаёт временный `DiGraph` только из
parent-child relations и проверяет, что он является DAG.

NetworkX-граф не сохраняется и не является источником истины. После проверки его
можно удалить: все исходные связи остаются в `Genealogy`.

В renderer NetworkX также используется для:

- connected components брачных связей;
- component DAG;
- topological sorting;
- descendants;
- barycentric ordering.

## 8. Render projection

Renderer преобразует domain-модель во внутреннее представление:

- `PersonBox` — размеры текстовой подписи;
- `PartnerComponent` — горизонтальный блок связанных браками людей;
- `FamilyView` — родители, дети и точка выхода ветви;
- `PersonPosition` — абсолютная геометрия человека.

Вспомогательные модели находятся в `infrastructure/graph/models.py`, а
формирование и измерение подписей — в `labels.py`.

## 9. Layout

Сначала супруги объединяются в connected components. Человек с несколькими
браками остаётся одним узлом внутри общего горизонтального компонента.

Затем строится DAG компонентов. Уровень компонента соответствует поколению.

Для простого дерева используется depth-contour layout. Контуры описывают реально
занятую ширину поддерева на каждой глубине, поэтому короткая ветвь не резервирует
пустое место до низа страницы.

Для DAG с несколькими входящими компонентами используется послойная раскладка с
barycenter ordering и несколькими проходами упаковки.

После общей раскладки единственный ребёнок при возможности выравнивается под
центром родителя или брачной линии. Вместе с ним сдвигается всё поддерево.

## 10. Graphviz

Graphviz используется только как vector-output backend.

Приложение передаёт каждому узлу фиксированную позицию:

```text
pos="x,y!"
pin="true"
```

Рендер выполняется через `neato -n2`, поэтому Graphviz не пересчитывает layout.

Каждая видимая связь строится вручную из горизонтальных и вертикальных
сегментов. Диагональный сегмент отвергается исключением.

Точки стыков представлены невидимыми anchor nodes. Горизонтальная шина
разрезается в каждой точке вертикального присоединения, а соседние сегменты
используют один и тот же anchor. Это устраняет выступы и зазоры в углах и
T-стыках.

## 11. PDF

Каждая родословная сначала рендерится во временную PDF-страницу. Затем
`PdfBookComposer`:

1. выбирает A5 или A4 landscape;
2. вычисляет допустимый scale;
3. центрирует исходную страницу;
4. объединяет страницы в одну книгу.

Временные файлы удаляются вместе с `TemporaryDirectory`.

## 12. Где сохраняются связи

Во время запуска связи находятся только в `Genealogy`:

```text
genealogy.marriages
genealogy.parent_child_relations
genealogy.family_child_relations
```

NetworkX и Graphviz получают временные производные представления. После
завершения процесса объекты Python освобождаются. В PDF/SVG остаётся только
визуальный результат, а не полноценная domain-модель.

## 13. Почему нет базы данных

Текущий сценарий:

- один небольшой Excel;
- один пакетный запуск;
- отсутствие параллельного редактирования;
- отсутствие запросов к данным между запусками;
- Excel остаётся источником истины.

База данных потребуется только при появлении веб-редактора, истории изменений,
нескольких пользователей, ручного CRUD или долгоживущего API.
